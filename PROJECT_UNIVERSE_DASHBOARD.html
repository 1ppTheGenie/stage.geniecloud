‚òÅÔ∏è GenieCloud - The Content Engine CRITICAL INFRASTRUCTURE

GenieCloud is the heart of content rendering. It takes XML data + XSL templates and transforms them into HTML pages, PDF brochures, and marketing assets. All content flows through GenieCloud. Understanding this system is essential for building new features.

üèóÔ∏è Architecture Overview

Component Location Purpose
XSL Templates public/_assets/_xsl/ XSLT stylesheets that transform XML into HTML/SVG. Templates for landing pages, flyers, brochures, etc.
CSS Themes public/_assets/themes/ Theme files like compass.css. Use theme="compass" + themeHue="dark" (not compass-dark.css)
Genie API genie-api/src/ Node.js Lambda functions that handle rendering, PDF generation, and S3 uploads
Genie Processor genie-processor/ XSLT transformation engine (has module issues locally - use post-processing approach)
S3 Bucket genie-cloud (us-west-1) Production hosting for all rendered content. AWS Profile: genie-hub-active

üß™ Key Learnings from 12/25/2025 Session

  • XML ‚Üí XSL ‚Üí HTML: The rendering pipeline takes XML data, applies XSL templates, outputs HTML/PDF. All content uses this flow.
  • Spoofing for Widgets: To get live widget data, set userId in XML to an agent with active MLS data. Display info comes from agent block, API calls use userId.
  • Theme Configuration: Use theme="compass" with themeHue="dark". There's no separate compass-dark.css file.
  • Brochure Generation: PDF brochures are auto-generated when pages render through genie-api. Manual deploys bypass this - need to trigger separately.
  • Post-Processing Approach: For manual deploys, download rendered HTML, fix URLs/encoding with Python, upload to S3. Works around local XSLT issues.
  • Encoding Issues: UTF-8 characters (smart quotes, √Ç) need post-processing. Use Python to replace before deploying.
